# Задание на создание инструментов для LLM

В классы реализации инструментов сохраняем в папке `src/tools`. Все файлы с реализацией имеют постфикс Tool

При проектировании, все ограничения и настройки инструмента должны быть вынесены в свойства инструмента и их можно было бы менять.

Пример @vendor/neuron-core/neuron-ai/src/Tools/Tool.php Интерфейс инструмента @vendor/neuron-core/neuron-ai/src/Tools/ToolInterface.php

Примеры инструментов в папке vendor/neuron-core/neuron-ai/src/Tools/Toolkits

---

### 1. Инструмент **Glob** (поиск файлов по шаблону)

**Назначение:**  
Находит файлы и директории, соответствующие заданному шаблону (как команда `find` или glob-паттерны в командной оболочке). Используется для быстрого получения списка релевантных файлов, когда модель должна работать с определённой группой файлов (например, «все тестовые файлы», «все файлы с определённым расширением»).

**Алгоритм работы:**

1. **Формирование запроса:** Модель (на основе промпта) решает, что необходимо найти файлы по некоторому критерию. Она вызывает инструмент `GlobTool`, передавая параметр `pattern` (например, `"src/**/*.test.js"`).
2. **Проверка разрешений:** Обычно `Glob` считается безопасным (только чтение).
3. **Выполнение поиска:**
   - запускает внутреннюю функцию, которая обходит файловую систему, начиная с текущей рабочей директории (или корня проекта).
   - Используется стандартный механизм glob (например, `fast-glob` или встроенный `glob` модуль Node.js) с поддержкой `**`, `*`, `?` и т.д.
   - Поиск рекурсивный, но может быть ограничен настройками (исключение `.git`, `node_modules` по умолчанию).
4. **Фильтрация и сортировка:** Найденные пути приводятся к относительным (относительно корня проекта) и сортируются для консистентности.
5. **Возврат результата:**
   - Возвращается список строк (путей) в формате JSON или текстовом (каждый путь на новой строке).
   - Если файлов слишком много (превышен лимит, заданный в настройках, например 1000), результат может быть обрезан с предупреждением.
6. **Обработка моделью:** Модель получает список файлов и использует его для принятия решений (например, затем вызывает `View` для каждого или `Grep` для поиска внутри них).

**Особенности:**
- Поиск не следует символическим ссылкам (по умолчанию) для безопасности.
- Может учитывать `.gitignore` (если настроено).
- Используется для сбора контекста перед более сложными действиями.

### 2. Инструмент **Grep** (поиск текста внутри файлов)

**Назначение:**  
Ищет заданный текст или регулярное выражение внутри содержимого файлов. Аналог команды `grep -r`. Позволяет модели быстро находить места использования функций, переменных, определений классов и т.д.

**Алгоритм работы:**

1. **Формирование запроса:** Модель передаёт параметры: `pattern` (строка или regex) и опционально `path` (путь к файлу/директории для поиска) или `include`/`exclude` (шаблоны файлов).
2. **Проверка разрешений:** Как чтение, обычно разрешения не требует.
3. **Выполнение поиска:**
   - Система сканирует файлы, соответствующие указанным путям (если не задано — весь проект).
   - Для каждого файла читается его содержимое (по частям, если файл большой).
   - Применяется поиск по регулярному выражению (или точному совпадению) к каждой строке.
   - Может использоваться оптимизированная библиотека (например, `ripgrep` внутри, если установлен, или встроенный токенизированный поиск).
4. **Сбор результатов:**
   - Для каждого совпадения сохраняется: путь к файлу, номер строки, сама строка (или её фрагмент).
   - Результаты группируются по файлам.
5. **Возврат результата:**
   - Возвращается структурированный список (например, массив объектов) или текстовый отчёт в формате, похожем на вывод `grep -n`.
   - При большом количестве совпадений может быть применено сокращение (например, первые 50 совпадений).
6. **Обработка моделью:** Модель анализирует найденные вхождения и может решить, какие файлы нужно изучить подробнее с помощью `View`.

**Особенности:**
- Поддержка regex с учётом особенностей JS (или используемой библиотеки).
- Игнорирование бинарных файлов.
- Возможность искать только в файлах определённых типов (например, `*.js`).

### 3. Инструмент **Bash** (выполнение shell-команд)

**Назначение:**  
Позволяет модели выполнять произвольные команды в терминале пользователя. Это самый мощный (и потенциально опасный) инструмент, так как даёт доступ к системе.

**Алгоритм работы:**

1. **Формирование запроса:** Модель решает, что нужно выполнить команду, и вызывает `BashTool` с параметром `command` (строка команды). Может также передавать `timeout` или `working_dir`.
2. **Проверка разрешений:**
   - В настройках инструмента должно быть разрешение на определённые команды по шаблону.
   - Также может быть ограничение по времени выполнения.
3. **Выполнение команды:**
   - После получения разрешения, система запускает команду в дочернем процессе с использованием shell (по умолчанию `/bin/sh` или оболочка пользователя).
   - Устанавливается рабочий каталог (обычно корень проекта, если не указано иное).
   - Настраивается окружение (переменные, наследуемые от родительского процесса).
4. **Мониторинг выполнения:**
   - Потоки stdout и stderr захватываются в реальном времени.
   - Если команда выполняется дольше заданного таймаута (по умолчанию может быть 30 секунд), процесс принудительно завершается.
5. **Обработка результата:**
   - После завершения (успех, ошибка, таймаут) stdout, stderr и код возврата собираются в единый отчёт.
   - Если вывод слишком большой, он обрезается с уведомлением.
6. **Возврат модели:**
   - Модели передаётся объект с полями: `stdout`, `stderr`, `exit_code`, `timed_out` (если был таймаут), `command` (выполненная команда).
   - Модель анализирует результат и решает следующие шаги (например, если ошибка — исправить код и перезапустить).

**Особенности:**
- Возможность запускать интерактивные программы ограничена (не поддерживается полноценный псевдотерминал, поэтому сложные редакторы вряд ли запустятся корректно).
- Для длительных процессов (серверы) может быть ограничение по времени.
- Модель может комбинировать несколько команд, но каждая требует отдельного разрешения.

### 4. Инструмент **View** (чтение содержимого файла)

**Назначение:**  
Предоставляет модели содержимое одного или нескольких файлов. Используется для получения контекста, когда модель должна понять код, документацию, данные.

**Алгоритм работы:**

1. **Формирование запроса:** Модель передаёт параметр `path` (путь к файлу) или массив путей. Может также указывать диапазон строк (например, `lines: 10-20`) или `max_lines` для ограничения объёма.
2. **Проверка разрешений:** Чтение файлов обычно разрешено без подтверждения (но может быть настроено иначе).
3. **Чтение файла:**
   - Система проверяет, существует ли файл, доступен ли для чтения, не превышает ли размер допустимый лимит (например, 1 МБ).
   - Файл читается синхронно или асинхронно (в зависимости от реализации) в кодировке UTF-8.
   - Если файл слишком большой, он может быть прочитан частично (первые N строк/байт) с предупреждением.
4. **Форматирование вывода:**
   - Возвращается содержимое файла, обычно с номерами строк (чтобы модель могла ссылаться на них при редактировании).
   - Если запрошен диапазон строк, возвращается только этот диапазон.
   - При чтении нескольких файлов они возвращаются в структурированном виде (например, разделители с именем файла).
5. **Возврат модели:**
   - Модель получает текст файла (или нескольких). Затем она может использовать эту информацию для анализа, генерации изменений или ответа на вопрос.

**Особенности:**
- Автоматическое обнаружение синтаксиса для подсветки (если включено) — может помочь модели лучше понять код.
- Учитываются настройки игнорирования (например, бинарные файлы могут быть пропущены с предупреждением).
- Часто используется вместе с `Glob` или `Grep`, когда модель сначала находит релевантные файлы, а потом просматривает их содержимое.

### 5. Инструмент **Edit** (редактирование файла)

**Назначение:**  
Позволяет модели вносить изменения в существующий файл (или создавать новый). Это основной инструмент для автоматического рефакторинга, исправления ошибок, добавления кода.

**Алгоритм работы:**

1. **Формирование запроса:** Модель передаёт параметры: `path` (путь к файлу), `old_string` (фрагмент, который нужно заменить) и `new_string` (чем заменить). Либо может быть передан весь новый контент (режим полной замены).
2. **Проверка разрешений:**
   - Изменение файлов требует обязательного подтверждения пользователя (по умолчанию).
   - Пользователь видит дифф (изменения) и запрос на подтверждение. Можно разрешить или отклонить.
3. **Валидация и поиск контекста:**
   - Система читает текущее содержимое файла.
   - Ищет вхождение `old_string` в файле. Должно быть **ровно одно** совпадение (если больше — ошибка, если не найдено — ошибка). Это предотвращает случайные множественные замены.
   - Опционально может выполняться нечёткий поиск (fuzzy matching) для устойчивости к небольшим изменениям (например, пробелы), но обычно требуется точное совпадение.
4. **Выполнение замены:**
   - Если `old_string` найден, он заменяется на `new_string`.
   - Новое содержимое записывается во временный файл для проверки, затем атомарно заменяет оригинал (чтобы избежать повреждения при сбое).
5. **Пост-обработка:**
   - Может автоматически форматировать код (если в проекте есть Prettier или аналоги, и это настроено).
   - Возможно создание backup-копии (если включено в настройках).
6. **Возврат результата:**
   - Модели возвращается подтверждение об успешной замене (или ошибка).
   - Также может быть возвращён дифф изменений для информирования модели.

**Особенности:**
- Модель часто использует `View` перед `Edit`, чтобы получить текущий контекст и правильно сформировать `old_string`.
- Для создания нового файла используется отдельный режим `Write` (или параметр, указывающий, что файла нет).
- Поддерживается отмена изменений (undo) через историю в интерфейсе Claude Code.
- Важно, что `old_string` должен быть уникальным и достаточно длинным, чтобы избежать коллизий. Модель может включать окружающие строки для точности.
