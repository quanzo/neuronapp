## Файлы и вложения

Этот документ описывает, как проект работает с файлами контекста и вложениями для LLM.

### Вложения из консоли (`--file/-f`)

Консольные команды (`simplemessage`, `todolist`) позволяют передавать файлы через опцию `--file` (`-f`). Обработка реализована в `AttachmentHelper::buildAttachmentsFromPaths()`:

- на вход подаётся список путей (строки);
- относительные пути интерпретируются относительно текущей рабочей директории (`getcwd()`);
- для каждого пути выполняется проверка существования, типа файла и прав чтения;
- в случае ошибки (пустой путь, файл не найден/недоступен) в вывод пишется сообщение `<error>…</error>`, а метод возвращает `null`.

Тип вложения определяется по расширению и превращается в DTO:

- изображения (`png`, `jpg`, `jpeg`, `gif`, `webp`, `bmp`) → `ImageFileAttachmentDto`;
- текстовые файлы (`txt`, `md`, `log`, `json`, `yaml`, `yml`, `xml`, `csv`, `ini`, `conf`, `cfg` и файлы без расширения) → `TextFileAttachmentDto`;
- остальные расширения по умолчанию также считаются текстовыми.

Перед отправкой в LLM вложения проходят через `AttachmentHelper::deduplicateAttachments()`, чтобы исключить дубли (по типу, пути и метаданным).

### @‑ссылки в тексте skills/todolist

Внутри skills и todolist можно ссылаться на файлы контекста, используя синтаксис `@relative/path/to/file`:

- ссылки извлекаются функцией `FileContextHelper::extractFilePathsFromBody()`:
  - `@` в начале строки или после пробела/табуляции;
  - путь идёт до первого пробела/табуляции или конца строки;
  - результат — уникальный список относительных путей;
- для каждого пути `AttachmentHelper::buildContextAttachments()`:
  - получает настройки `context_files` из `ConfigurationApp::get()` (из `config.jsonc`);
  - проверяет соответствие `allowed_paths`/`blocked_paths` (glob‑маски, обрабатываемые через `fnmatch`);
  - использует `DirPriority::resolveFile()` для поиска файла в приоритетных директориях;
  - учитывает общий лимит размера (`max_total_size`), чтобы не перегружать контекст.

Возвращаемая структура:

```php
[
  'attachments' => list<AttachmentDto>,
  'totalSize'   => int,
]
```

Эти вложения добавляются:

- для skills — в `Skill::execute()` при формировании запроса к агенту;
- для todolist — отдельно для каждого todo в `TodoList::execute()`, чтобы каждый шаг мог иметь свой контекст.

### Настройки `context_files` в `config.jsonc`

В разделе `context_files` файла `config.jsonc` задаются правила подключения файлов по @‑ссылкам:

- `context_files.enabled` (`bool`) — включить/выключить механику;
- `context_files.max_total_size` (`int`, байты) — общий лимит размера всех подключённых файлов;
- `context_files.allowed_paths` (`string|array`) — список glob‑масок разрешённых путей;
- `context_files.blocked_paths` (`string|array`) — список масок запрещённых путей.

Пример:

```jsonc
{
  "context_files": {
    "enabled": true,
    "max_total_size": 1048576,
    "allowed_paths": [
      "docs/*",
      "src/**/*.php"
    ],
    "blocked_paths": [
      "vendor/**",
      "node_modules/**",
      "temp/**"
    ]
  }
}
```

Если `enabled = false` или лимит размера неположительный, `AttachmentHelper::buildContextAttachments()` всегда возвращает пустой список вложений.

### Где размещать файлы контекста

Рекомендуемый подход:

- документацию, которая может подключаться как контекст, хранить в `docs/` или `testapp/docs/`;
- исходный код, который нужно редко (и адресно) подгружать, разрешать через `allowed_paths` с масками `src/**/*.php`;
- явно блокировать большие или чувствительные директории (`vendor/**`, `node_modules/**`, временные каталоги).

Такой подход позволяет:

- явно контролировать объём и источник контекста для LLM;
- избегать случайной утечки лишних данных;
- повторно использовать справочные `.md`‑файлы из `docs/` как часть запроса.
